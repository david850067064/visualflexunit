Packagecom.allurent.flexunit2.vfu.framework.testsequence
Classpublic class TestSequenceManager

This class provides much of VfuTestCase's functionality. It is stored in that class's testSequence property, and implements test sequences by creating and executing a queue of TestSequenceCommand instances. We've tried to comment it fairly thoroughly for two reasons: a) because it is so central to VFU's functionality, and b) because it is fairly complex. These ASDocs only display comments for public members of the class. If you're interested in an in depth look at how the class works we suggest that you look at the source code where you'll find not only the code but comments explaining private members, etc.

addAssertComponentMatchBaseline()method 
public function addAssertComponentMatchBaseline(comp:UIComponent, testID:int, timeout:int, judgmentStrategy:IBitmapMatchJudge = null):void

This method adds an AssertComponentMatchBaseline command instance to the test sequence queue. You should add it after you've manipulated your component as desired and called addRefresh(). You can repeat this sequence - manipulate-refresh-assert - multiple times in your test method.

Parameters
comp:UIComponent — The UIComponent instance whose appearance you are comparing to a saved baseline.
 
testID:int — Every call to addAssertComponentMatchBaseline() must pass in an integer testID that is unique within the testmethod. This is used to name the baseline file that is used by the assert.
 
timeout:int — Optional - Pass in an integer specifying milliseconds if you wish this command to allow more or less time to pass before throwing an error. The default is 10 seconds.
 
judgmentStrategy:IBitmapMatchJudge (default = null) — Optional - If you'd like VFU to use some logic other than its default logic to decide whether a test result is close enough to its saved baseline you can write a class that implements IBitmapMatchJudge, instanciate it, and pass it in here.
addEventDispatch()method 
public function addEventDispatch(target:EventDispatcher, event:Event):void

This method adds a DispatchEvent command object to the test sequence queue which, when executed calls dispatch() event on the target you specify. We use this to simulate mouse and keyboard actions using MouseEvents and KeyboardEvents and a UIComponent target, but this method could be used for other events with other types of EventDispatcher targets. Let us know what creative uses you come up with!

Parameters
target:EventDispatcher — The EventDispatcher instance that you'd like to dispatch the event from.
 
event:Event — The Event you'd like to dispatch.

addEventOrQuiesceWait()method 
public function addEventOrQuiesceWait(target:EventDispatcher, eventType:String, timeout:int, quiesceDuration:int):void

This method adds a WaitForEventOrQuiesce command object to the test sequence queue. As its name implies this command waits for either of:

a) The arrival of an event, in either its capture or target phase, or

b) The passage of a period of time without that event happening.

This method is used as one of several steps set up by addRefresh() to create an opportunity for a UIComponent to render fully before its appearance is tested. See the comments in WaitForEventOrQuiesce for details on why its behavior is useful. It's not clear to us whether it will also be useful to the writers of visual tests. If it is, let us know. :)

Parameters
target:EventDispatcher — The EventDispatcher instance that you'd like to listen to for the specified event dispatch.
 
eventType:String — A string specifying the type of event you'd like to listen for.
 
timeout:int — Optional - Pass in an integer specifying milliseconds if you wish this command to allow more or less time to pass before throwing an error. The default is 10 seconds.
 
quiesceDuration:int — Optional - The amount of "quiet time" you'd like your test sequence to wait for, in milliseconds. The default is 300.

public function addQuiesceWait(timeout:int, quiesceDuration:int):void

This method adds a WaitForQuiesce command object to the test sequence queue. As its name implies this command waits for the passage of a period of time without an UPDATE_COMPLETE event being received by the component display panel.

This method is used as one of several steps set up by addRefresh() to create an opportunity for a UIComponent to render fully before its appearance is tested.

Parameters
timeout:int — Optional - Pass in an integer specifying milliseconds if you wish this command to allow more or less time than the default to pass before throwing an error. The default is 10 seconds.
 
quiesceDuration:int — Optional - The amount of "quiet time" you'd like your test sequence to wait for, in milliseconds. The default is 300.
addRefresh()method 
public function addRefresh(timeout:int, quiesceDuration:int):void

This method adds several different command objects to the test sequence queue which, working together, will hopefully give a displayed UI component an adequate opportunity to render fully. Put another way, what we're trying to accomplish here is reliable full rendering. As this project is currently in its 'experimental alpha' stage, it is possible that the approach currently used will need to be tweaked. It appears to work fine, but we invite your feedback.

Parameters
timeout:int — Optional - Pass in an integer specifying milliseconds if you wish this command to allow more or less time to pass before throwing an error. The default is 10 seconds.
 
quiesceDuration:int — Optional - The amount of "quiet time" you'd like your test sequence to wait for, in milliseconds, as part of the refresh process. The default is 300.

commandFinished()method 
public function commandFinished(command:TestSequenceCommand, failureError:Error = null):void

This method is called by the current active command when it's finished. It passes in an instance of itself and, optionally, an Error instance. The Error instance, if any, does two things: a) it signals an end of the test sequence, even if some unexecuted commands remain in the queue, and b) it ends up in addAsyncHandler() where it is thrown. See the note in addAsyncHandler()'s comments for an explanation of why it is important for errors to be thrown there.

Parameters
command:TestSequenceCommand — The currently running TestSequenceCommand which called this method.
 
failureError:Error (default = null) — See explanation above
